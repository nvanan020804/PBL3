Khi sử dụng SQL Server Authentication trong một chuỗi connection string, bạn không nên điền mật khẩu trực tiếp vào file cấu hình như trong đoạn bạn đưa ra (Password=***********). Việc điền mật khẩu trực tiếp trong file cấu hình có thể dẫn đến vấn đề bảo mật, đặc biệt khi file cấu hình bị lộ hoặc được kiểm soát bởi nhiều người.

Thay vào đó, bạn có thể làm theo một số cách sau để bảo mật mật khẩu:

  1. Sử dụng Secret Manager (ASP.NET Core): Bạn có thể sử dụng công cụ Secret Manager trong ASP.NET Core để lưu trữ mật khẩu trong môi trường phát triển mà không cần đưa chúng vào trong các file cấu hình. Điều này giúp bảo mật mật khẩu trong quá trình phát triển.

    Để cấu hình Secret Manager, bạn có thể sử dụng lệnh:

    bash
    dotnet user-secrets set "ConnectionStrings:WebShop" "Data Source=VAN_AN\\MSSQLSERVER12;Initial Catalog=DOTNET;Persist Security Info=True;User ID=sa;Password=your_password_here;Encrypt=True;Trust Server Certificate=True"

  2. Sử dụng môi trường (Environment Variables): Bạn có thể sử dụng biến môi trường để lưu mật khẩu. Điều này cũng giúp bảo mật thông tin khi triển khai ứng dụng lên các môi trường khác nhau (local, staging, production). Ví dụ trong Windows:

    bash
    setx CONNECTIONSTRINGS__WEBSHOP "Data Source=VAN_AN\\MSSQLSERVER12;Initial Catalog=DOTNET;Persist Security Info=True;User ID=sa;Password=your_password_here;Encrypt=True;Trust Server Certificate=True"

    Sau đó trong appsettings.json, bạn có thể truy xuất chuỗi kết nối này mà không cần phải lưu mật khẩu trực tiếp.

  3. Sử dụng Azure Key Vault (cho các ứng dụng triển khai trên Azure): Nếu ứng dụng của bạn được triển khai trên Azure, bạn có thể lưu mật khẩu trong Azure Key Vault và truy xuất nó khi cần.

Bằng cách này, bạn có thể giữ thông tin mật khẩu an toàn hơn và tránh việc để mật khẩu trong các file cấu hình công khai.

Ok, để test database ở lớp bằng laptop, bạn cần chuẩn bị vài thứ trước:

1. Cài SQL Server trên laptop

  • Nếu bạn chưa cài, nên cài SQL Server Express hoặc SQL Server Developer Edition (miễn phí).

  • Cài thêm SQL Server Management Studio (SSMS) để dễ thao tác với database.

2. Tạo database để test

  • Mở SSMS → kết nối vào SQL Server → click phải vào Databases → chọn New Database → tạo một database test, ví dụ: TestDB.

3. Sửa lại Connection String

Giả sử bạn cài SQL Server trên máy mình (localhost), thì connection string mẫu sẽ như:

json
"ConnectionStrings": {
  "WebShop": "Server=localhost\\SQLEXPRESS;Database=TestDB;User Id=sa;Password=your_password;TrustServerCertificate=True;"
}

  • Nếu instance name là mặc định thì chỉ cần localhost.

  • Nếu instance riêng, thì ghi localhost\\TênInstance.

Chú ý:

  • Nếu bạn đang dùng Windows Authentication (không dùng sa), connection string sẽ không cần User Id và Password.

  • Ví dụ dùng Windows Authentication:

    json
    "ConnectionStrings": {
  "WebShop": "Server=localhost\\SQLEXPRESS;Database=TestDB;Trusted_Connection=True;TrustServerCertificate=True;"
}

4. Mở SQL Server cho phép remote connection (nếu cần)

  • Nếu bạn muốn 2 máy trong lớp kết nối với nhau thì phải bật remote access và mở port 1433 (nhưng thường test một mình trên laptop thì không cần).

5. Test bằng Project ASP.NET Core

  • Trong ASP.NET Core, trong Program.cs hoặc Startup.cs, đảm bảo bạn đọc connection string đúng:

csharp
builder.Services.AddDbContext<MyDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("WebShop")));

  • Sau đó run thử để check xem có lỗi connection hay scaffold được chưa.

6. Trường hợp cần fake database

  • Nếu không muốn setup SQL Server, bạn có thể dùng SQL Server LocalDB (có sẵn khi cài Visual Studio) cho tiện test: ConnectionString LocalDB ví dụ:

json
"ConnectionStrings": {
  "WebShop": "Server=(localdb)\\mssqllocaldb;Database=TestDB;Trusted_Connection=True;"
}

  • LocalDB là database nhẹ, tự động tạo file .mdf ngay trong máy.
